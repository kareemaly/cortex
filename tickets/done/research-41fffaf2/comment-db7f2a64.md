## Summary

Researched and documented all differences between Claude Code and OpenCode agent defaults in Cortex.

### Key Findings

1. **Prompts are 100% identical** — All 16 prompt files (architect, meta, ticket/{work,debug,research,chore} x {SYSTEM,KICKOFF,APPROVE}) are byte-for-byte the same between both agent types.

2. **cortex.yaml diverges significantly** — Claude Code has carefully scoped CLI args per session type (permission mode, tool allowlists, dangerous-permissions bypass). OpenCode has zero args configured for any role.

3. **Lifecycle hooks are Claude-only** — Three hooks (PostToolUse, Stop, PermissionRequest) enable real-time agent status tracking in the TUI. OpenCode is explicitly skipped in spawn code, so the TUI can't show agent state.

4. **Permission model differs** — Claude Code uses `--allowedTools` to pre-approve specific MCP tools per role. OpenCode runs with blanket `bypassPermissions` mode and `"*": "allow"`.

5. **Runtime spawning differs** — Different binaries, config delivery mechanisms (files vs env var), and prompt injection. OpenCode config is generated by transforming Claude-format MCP config at runtime via `opencode_config.go`.

### Recommendations

1. **Investigate OpenCode CLI args** — Determine if OpenCode supports equivalent permission/allowlist flags and populate them in defaults.
2. **OpenCode hooks** — If OpenCode supports any hook/callback mechanism, wire it up for agent status tracking in the TUI.
3. **Document the permission model gap** — Users choosing OpenCode should understand it runs with broader permissions by default.
4. **Session resume** — Verify if OpenCode supports session resume and wire it into the launcher.
